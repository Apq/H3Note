## 1. 目标

围绕英雄无敌 3 SoD、HD Mod 框架及其插件做可复现的静态/动态逆向：

- 快速确认工具装在哪儿；
- 明确 radare2 / x32dbg / IDA 各自用途；
- 区分 `Heroes3.exe` 本体、HD Mod 包装层、插件 DLL；
- 把已确认成果沉淀到专题逆向笔记中，避免重复试错。

## 2. 工具位置

```text

```

命令行也可直接找到：

### 常用工具分工

| 工具 | 主要用途 |
|------|----------|
| radare2 | 快速反汇编、查调用者、批量验证地址、生成函数视图 |
| x32dbg | 动态调试、下断点、看寄存器/栈/对象指针/内存字段 |
| IDA Free | 长链路静态阅读、重命名函数、整理结构体 |
| Cheat Engine | 快速找运行时数值、验证字段偏移 |
| Process Explorer / Process Hacker | 看进程、模块、句柄、权限 |
| VS Code / Notepad++ | 维护逆向笔记、地址表、Hook 方案 |

> `Heroes3.exe` 是 32 位程序，动态调试应使用 32 位调试器，例如 x32dbg。

## 3. 分析对象先分层

逆向前先确认目标属于哪一层：

| 层 | 典型内容 | 优先看哪里 |
|----|----------|------------|
| SoD 本体 | 战斗 AI、UI、存读档、资源管理、战斗逻辑 | `Heroes3.exe` |
| HD Mod 框架 | 分辨率、窗口包装、消息转发、兼容层、HD buffer | HD Mod DLL / 配置 / 包装入口 |
| 插件 | H3API/ERA/HD 插件 Hook、补丁逻辑、崩溃修复 | 插件源码 + Hook 地址 |
| 资源系统 | LOD、PCX、DEF、资源名限制、加载覆盖 | `Data\`、LOD、资源管理器 |

经验规则：

- 找战斗 AI、战斗对象、原版 UI 构造，优先看 `Heroes3.exe`；
- 找高分辨率、窗口位置、HD buffer、ZCN2 崩溃，必须考虑 HD Mod；
- 找插件行为，先看插件源码，再回到 Hook 地址反查原函数；
- 不要把战略 AI、战场 AI、UI、HD 包装层混成一个问题。

## 4. radare2 基本用法

以下以当前常用目标为例：

```text
D:\Heroes3\Heroes3_2026.05.01\Heroes3.exe
```

### 打开并设置基址

```powershell
D:\Programs\radare2-6.1.6-w64\bin\radare2.exe -B 0x400000 D:\Heroes3\Heroes3_2026.05.01\Heroes3.exe
```

进入后常用：

```r2
aaa              # 全分析，约几十秒
s 0x469f30       # 跳到地址
pdf              # 反汇编当前函数
pd 80            # 从当前位置反汇编 80 条指令
axt 0x469f30     # 查谁引用/调用这个地址
afll~469f30      # 在函数列表中过滤地址或名字
izz~CrStkP1      # 查字符串
pxw 64 @ 0x6747b0 # 看内存/数据，按 word/dword 展示
```

### 一次性命令

适合快速查函数或导出片段：

```powershell
D:\Programs\radare2-6.1.6-w64\bin\radare2.exe -B 0x400000 -A -q -c "s 0x469f30; pdf" D:\Heroes3\Heroes3_2026.05.01\Heroes3.exe
```

查调用者：

```powershell
D:\Programs\radare2-6.1.6-w64\bin\radare2.exe -B 0x400000 -A -q -c "axt 0x469f30" D:\Heroes3\Heroes3_2026.05.01\Heroes3.exe
```

## 5. 常见逆向流程

### 静态确认

1. 用 radare2 / IDA 定位函数入口、调用者、关键分支；
2. 标出参数来源、寄存器含义、结构体偏移；
3. 只把能由反汇编直接支持的结论记为“确认”；
4. 不确定的推测先写在临时草稿，不写入正式逆向笔记。

### 动态验证

1. 用 x32dbg 附加或启动 `Heroes3.exe`；
2. 在候选地址下断点；
3. 触发游戏行为，例如右键、战斗回合、存档、施法、打开窗口；
4. 记录当时的 `ECX/ESI/EDI/ESP`、参数、对象指针、字段值；
5. 用多次触发排除偶然路径。

### 插件 Hook 验证

1. 先确认 Hook 地址是不是函数边界；
2. 函数边界优先用 HiHook；函数内部指令点才用 LoHook；
3. thiscall 要确认 `ECX=this` 和隐藏参数；
4. 改 Hook 类型后必须实测进入、返回、关闭窗口/战斗结束等生命周期路径；
5. 崩溃修复类 Hook 要记录原崩溃点、调用链、触发条件、修复策略。

## 6. HoMM3 / HD Mod 重点经验

### 战斗 AI

追踪顺序建议：

1. AI 回合入口；
2. 决策主函数；
3. 候选目标/候选动作生成；
4. 分数写入点；
5. 比较/排序函数；
6. 最终执行攻击、移动、施法的调用点。

不要只因为某字段被读取，就下结论“AI 按该字段选目标”。应确认：

- 谁生成分数；
- 分数由哪些字段组成；
- 谁比较两个分数；
- 最终执行路径是否真的使用该排序结果。

### UI / 对话框

记录时优先写清：

- 窗口对象指针在哪个寄存器；
- 控件列表位置；
- 控件 id、vtable、坐标、宽高、flags；
- 构造点、消息循环点、析构点；
- HD/ZCN2 是否会受窗口绝对坐标和 buffer 高度影响。

### 资源系统

资源相关结论应写入资源专题笔记。当前已确认的重要经验包括：

- HoMM3 资源名含扩展名最多 11 字符；
- PCX/DEF 加载路径和格式要区分；
- `Data\` 散文件覆盖 LOD 同名资源；
- 不要在指南里重复堆资源格式细节。

## 9. 最小工作原则

- 先确认目标层级，再开工具；
- 优先复用已有逆向笔记中的确认地址；
- 新结论必须能被反汇编、dump、断点或日志支撑；
- 指南保持短，逆向成果进专题笔记；
- 发现旧笔记错误时，直接在对应笔记中更正并说明依据。
