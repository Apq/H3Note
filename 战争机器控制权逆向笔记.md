## 0. 结论速览

- **控制权分流函数**：`FUN_00474520 @ 0x00474520`
  - 返回 `0` → 交给本地人类玩家输入（等待鼠标/键盘）
  - 返回 `1` → 走自动 / AI 代打（不等待本地输入）
- **最薄查询封装**：`FUN_004744d0 @ 0x004744d0`
- **统一动作执行出口**：`FUN_004786b0 @ 0x004786b0`（按 `battle+0x3c` = action code 落地）
- **总闸**：`FUN_0046a080 @ 0x0046a080`（回放/非本地交互对局标志，非 0 时短路人工输入）
- **接管首选点**：在 `FUN_004744d0` / `FUN_00474520` 返回值处让目标机器返回 1（走自动），并在 `FUN_004786b0` 之前写入期望的 `+0x3c/+0x40/+0x44`，复用原版回合推进/动画/重绘。

## 2. FUN_00474520 分支逻辑（判定核心）

```c

  case 0x91: // 投石车

    if (*(stack+0x132c4)!=0 && DAT_006987ec!=0) return 1;  // AI托管+全局开关 → 自动
    if (*(hero+0xd3) == 0) return 1;           // 无弹道术 → 自动
    break;
  case 0x92: case 0x95: // 弩车/箭塔
    if (*(stack+0x13d68) != 0) return 0;
    if (*(stack+0x132c4)!=0 && DAT_006987f0!=0) return 1;

    if (*(hero+0xdd) == 0) return 1;           // 无炮术 → 自动
    break;
  case 0x93: // 急救帐篷
    if (*(stack+0x13d68) != 0) return 0;
    if (*(stack+0x132c4)!=0 && DAT_006987f4!=0) return 1;
    if (*(hero+0xe4) == 0) return 1;           // 无急救术 → 自动
    break;
  default: // 普通部队

  }
  // 通用尾部：判断该侧是否人类玩家控制
  if (DAT_00691259==0 || *(stack+0x132b4)==0) {

  }

```

### 关键字段

| 偏移（相对 battle 基址） | 含义 |
|---|---|
| `+0x132c4` | AI 托管开关（"自动战斗"已开启）|
| `+0x13d68` | 观战/攻城布阵态标志（非 0 时战争机器强制人工）|
| `+0x132b4` | 是否本地交互（0=本地玩家在操作）|
| `+0x54a8 + side*4` | 该侧玩家索引 |
| `DAT_006987e4/ec/f0/f4` | 全局开关：普通部队 / 投石车 / 弩车箭塔 / 急救帐篷 的"AI 接管"总开关 |
| `DAT_00691259` | 快速/自动战斗全局标志 |

`FUN_004ce600(player)` = 判断玩家索引是否人类控制。

```c
       return (DAT_006987cc != 0);              // 本地：看 DAT_006987cc

```

- `DAT_00699538 + 0x1f69d` 是**回放/非本地交互对局**标志。非 0 表示当前不是真人交互（回放、快速战斗等），此时整条人工输入判定被短路。

## 5. 动作执行器 FUN_004786b0（统一出口）

按 `*(battle+0x3c)` = action code 分 case 落地：

| action code | 动作 | 关键调用 |
|---|---|---|
| 1 | 移动 WALK | FUN_005a0140 |
| 2 | 近战/行走攻击 | FUN_00445a30 |
| 3 | 防御 DEFEND | — |
| 4 | 等待 WAIT | — |
| 5 | 投降/魔法值结算 | — |
| 6 | 射击相关 | FUN_00478230 |
| 7 | 普通攻击 SHOOT | FUN_00445780 |
| 8 | 施法提示 | — |
| 9 | 投石车 CATAPULT | FUN_00445a50 |
| 0xa (10) | 怪物施法 | FUN_00447f80 |
| 0xb (11) | 急救治疗 FIRST_AID_TENT | FUN_004784c0 |
| 0xc (12) | 纯占位 NOTHING | — |

- 动作参数统一从 `battle+0x3c(action) / +0x40 / +0x44(target) / +0x48` 读取。
- 每个 case 结束调用 `FUN_00464500(+0x132b8, +0x132bc)` 刷新，`FUN_00475c20` 结束检查，末尾 `Combat_InitTurnOrder(1)` 推进回合、`Combat_InitCreatures()`。这是原版把"已决策动作"落到动画/重绘/回合推进的地方。

### 重要交叉印证：`+0x3c` == eBattleAction

`battle+0x3c` 的 action code 与 H3API `eBattleAction` 枚举（H3API.hpp 第2485行）完全一致：

```
CANCEL=0  CAST_SPELL=1  WALK=2  DEFEND=3  RETREAT=4  SURRENDER=5
WALK_ATTACK=6  SHOOT=7  WAIT=8  CATAPULT=9  MONSTER_SPELL=10
FIRST_AID_TENT=11  NOTHING=12
```

> 注意：反编译 case 里 1=移动/2=近战 的编号与 H3API 枚举（WALK=2/SHOOT=7）在个别项上因内部临时态略有出入，实现接管时以 **eBattleAction 枚举** 为准写入 `+0x3c`，并实测校验。

## 6. 接管方案建议（未实现）

### 最窄稳定 Hook 点

1. **判定点**：Hook `FUN_004744d0` 或 `FUN_00474520` 的返回值。
   - 当目标战争机器（0x91/0x92/0x93/0x95）满足 H3Auto 方案启用条件时，强制返回 1（走自动），使原版不再等待人类输入。
2. **动作注入点**：在 `FUN_004786b0` 执行前，写入 H3Auto 计算好的 `+0x3c(action) / +0x40 / +0x44(target)`。
3. 复用原版 `Combat_InitTurnOrder(1)` / 动画 / 重绘 / 状态记账，**不替换整个战斗循环**。

### 注意事项

- `+0x132c4`（AI 托管开关）与 `DAT_006987xx` 全局开关会直接改变 `FUN_00474520` 分支。接管逻辑要么复用这些开关，要么在 hook 里独立判断，避免与原版"自动战斗"耦合导致误触发。
- 接管必须在 `FUN_0046a080()` 返回假（真人交互局）前提下进行，回放/快速战斗不注入。
- `+0x13d68`（观战/攻城布阵态）非 0 时战争机器强制人工，接管时应尊重这一状态。

## 7. 参考坐标

- `H3CombatManager` 全局基址：`DAT_00699420`（别名 `DAT_006992d0` / `DAT_006992d4` / `DAT_00699538`）。
- 每堆栈 stride：`0x548`。
- `stacks[side][slot]` = `mgr + 0x54cc + (side*21 + slot)*0x548`。
- side 间距：`0x6ee8`（= 0x548 * 21），反编译中以 int 计步长时写作 `0x1bba`。
- 每侧 21 槽位，已含普通部队 + 战争机器 + 最多 3 个箭塔。
