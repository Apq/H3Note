## 1. 目标与原则

H3Auto 的自动化配置应拆成两个正交部分：

- **行动策略**：决定当前部队“做什么”。
- **目标策略**：决定行动“作用于哪支部队或哪个战场位置”。

必须支持：

1. 目标可选择**己方部队、敌方部队或战场位置**。
2. 行动不局限于攻击，还包括移动、防御、等待、急救帐篷疗伤、投石车攻击。
3. 攻击必须明确区分**近战攻击**和**远程攻击**。
4. H3Auto 主动执行失败时，可按配置决定是否**降级为防御**。
5. 除明确规定的战争机器回退场景外，普通部队绝不交给原版 AI。
6. 只生成动作和参数，由原版执行器完成动画、伤害、重绘、状态记账和回合推进；不得自行替换整个战斗循环。
7. 所有方案继续只保存在进程内存，不读写 INI。

## 2. 设置面板中的部队过滤规则

设置面板打开时，先读取当前玩家一方 `stacks[side][0..20]`，再按下表决定是否显示。

| 单位 | 是否进入设置列表 | 规则 |
|---|---:|---|
| 弹药车 `0x94` | 否 | 永远排除；无主动行动，始终保持原版处理 |
| 投石车 `0x91` | 有条件 | 英雄掌握弹道术时显示；无弹道术时排除，保持原版 AI |
| 弩车 `0x92` | 是 | 无论是否有炮术都显示；配置远程攻击时可由 H3Auto 接管 |
| 急救帐篷 `0x93` | 有条件 | **只有英雄掌握急救术时显示**；否则排除并保持原版处理 |
| 箭塔 `0x95` | 是 | 无论是否有炮术都显示；配置远程攻击时可由 H3Auto 接管 |
| 普通部队 | 是 | 始终显示 |

对应技能索引：

```cpp
BALLISTICS = 10; // 弹道术
ARTILLERY  = 20; // 炮术
FIRST_AID  = 27; // 急救术
```

### 急救帐篷最终规则

- 英雄**没有急救术**：帐篷不显示在设置面板中，H3Auto 不干预。
- 英雄**有急救术**：帐篷显示，可配置“手动”或“急救治疗”。
- 帐篷配置为手动：保持原版处理。
- 帐篷配置为急救治疗：H3Auto 按目标策略选择合法己方伤员并提交治疗动作。
- 帐篷不允许配置移动、近战、远程等无效动作；UI 直接隐藏这些选项。

## 3. 配置数据模型

当前 `action[5][21] + target[5][21] + downgrade[5][21]` 无法完整描述目标类型、阵营和选择算法，应升级为结构化规则。

```cpp
enum AutoActionKind : uint8_t {
    AA_MANUAL = 0,        // 不干预
    AA_DEFEND,            // 防御
    AA_WAIT,              // 等待
    AA_MOVE,              // 移动
    AA_MELEE_ATTACK,      // 近战攻击
    AA_RANGED_ATTACK,     // 远程攻击
    AA_FIRST_AID,         // 急救帐篷疗伤
    AA_CATAPULT           // 投石车攻击城墙
};

enum AutoTargetKind : uint8_t {
    AT_NONE = 0,          // 无目标
    AT_STACK,             // 部队目标
    AT_POSITION           // 战场位置目标
};

enum AutoTargetSide : uint8_t {
    ATS_OWN = 0,          // 己方
    ATS_ENEMY,            // 敌方
    ATS_EITHER            // 双方均可；执行前仍受动作合法性矩阵约束
};

enum AutoTargetSelector : uint8_t {
    SEL_FIXED = 0,        // 指定部队或指定位置
    SEL_RANDOM,           // 随机
    SEL_SEQUENTIAL,       // 按槽位/位置顺序轮换
    SEL_NEAREST,          // 距当前单位最近
    SEL_FARTHEST,         // 距当前单位最远
    SEL_RANGED_SPEED,     // 远程优先，其次速度高
    SEL_COUNT_HIGH,       // 数量最多
    SEL_COUNT_LOW,        // 数量最少
    SEL_MOST_WOUNDED      // 损失生命最多，主要用于急救帐篷
};

struct AutoTargetRule {
    AutoTargetKind kind;
    AutoTargetSide side;
    AutoTargetSelector selector;

    int8_t fixedSide;        // 固定部队目标：0/1
    int8_t fixedSlot;        // 固定部队目标：0..20
    int16_t fixedCreatureId; // 防止跨战斗误指向不同单位；不匹配则目标失效
    int16_t fixedHex;        // 固定位置目标：原版 hex 0..186
};

struct AutoStackRule {
    AutoActionKind action;
    AutoTargetRule target;
    bool allowDefendFallback;
};

AutoStackRule g_profiles[5][21];
int g_active_profile;
```

### 为什么固定部队不能只保存指针

`H3CombatCreature*` 只在当前战斗有效，跨战斗会失效。固定部队目标应保存：

```text
side + slot + creatureId 指纹
```

运行时重新从 `mgr->stacks[side][slot]` 解析，并验证：

- 槽位范围有效；
- 单位仍存活；
- creature id 与保存值一致；
- 当前动作允许该阵营作为目标。

验证失败即视为“没有合法目标”，进入防御降级逻辑。

### 位置的存储

位置始终保存原版 `stack.position` 使用的 battlefield hex（0..186），显示时继续使用现有的 17 列换算方式生成 `A01`–`K15` 标签。配置中不保存显示字符串。

## 4. 行动策略与原版动作

配置层的 `AutoActionKind` 与原版动作提交层分开，禁止直接把配置枚举值写入 `battle+0x3C`。

| 配置行动 | 原版语义 | 是否需要目标 |
|---|---|---:|
| 手动 | 保持原版人类输入 | 否 |
| 防御 | `DEFEND` | 否 |
| 等待 | `WAIT` | 否 |
| 移动 | `WALK` | 位置；部队目标可作为靠近锚点 |
| 近战攻击 | `WALK_ATTACK` | 敌方部队或有敌人的位置 |
| 远程攻击 | `SHOOT` | 敌方部队或有敌人的位置 |
| 急救治疗 | `FIRST_AID_TENT` | 己方伤员或有己方伤员的位置 |
| 投石车攻击 | `CATAPULT` | 合法城墙目标位置 |

原版动作字段位于 `H3CombatManager`：

```text
battle + 0x3C : action
battle + 0x40 : actionParameter
battle + 0x44 : actionTarget
battle + 0x48 : actionParameter2
```

但不同动作对四个字段的解释不同。实现时必须通过原版命令构建/验证路径生成参数，不可仅凭 H3API 枚举裸写动作值。

## 5. 行动与目标兼容矩阵

| 行动 | 己方部队 | 敌方部队 | 战场位置 | 无目标 |
|---|---:|---:|---:|---:|
| 手动 | — | — | — | 允许 |
| 防御 | — | — | — | 必须 |
| 等待 | — | — | — | 必须 |
| 移动 | 可作为靠近锚点 | 可作为靠近锚点 | 允许 | 不允许 |
| 近战攻击 | 不允许 | 允许 | 位置上有敌人时允许 | 不允许 |
| 远程攻击 | 不允许 | 允许 | 位置上有敌人时允许 | 不允许 |
| 急救治疗 | 允许，且必须受伤 | 不允许 | 位置上有合法己方伤员时允许 | 不允许 |
| 投石车攻击 | 不允许 | 不允许 | 仅合法城墙位置 | 不允许 |

UI 应根据当前行动动态缩小目标选项范围，避免用户配置明显无效的组合。

## 6. 目标选择器设计

统一解析函数：

```cpp
struct ResolvedTarget {
    H3CombatCreature* stack; // 部队目标，可为空
    int hex;                 // 最终目标格，-1 表示无位置
};

bool ResolveTarget(
    H3CombatManager* mgr,
    H3CombatCreature* self,
    const AutoStackRule& rule,
    ResolvedTarget& out);
```

### 6.1 部队目标候选集

先按 `target.side` 枚举双方 `stacks[2][21]`，再过滤：

- `numberAlive > 0`；
- 不是 `UNDEFINED`；
- 满足行动与阵营兼容矩阵；
- 满足动作专用合法性条件。

随后应用选择算法：

- `SEL_FIXED`：按 `side + slot + creatureId` 解析。
- `SEL_RANDOM`：从合法候选集中随机。
- `SEL_SEQUENTIAL`：按槽位顺序轮换；每个执行单位保存独立游标。
- `SEL_NEAREST / FARTHEST`：使用战场 hex 距离，而不是屏幕像素距离。
- `SEL_RANGED_SPEED`：远程单位优先，然后速度降序，再按等级/slot 稳定排序。
- `SEL_COUNT_HIGH / LOW`：按存活数量排序。
- `SEL_MOST_WOUNDED`：按损失总生命值降序；相同则优先当前存活数量多者。

所有排序都必须带稳定的最后比较项（如 `sideIndex`），保证同一战场状态下结果可重复。

### 6.2 位置目标

- `SEL_FIXED`：直接解析保存的原版 hex。
- `SEL_SEQUENTIAL`：按预设位置列表循环。
- 部队目标用于“移动”时，目标部队本身占据的格子不能作为移动终点；需要在目标周围寻找合法可达空格。

位置必须通过原版战场可达性/动作合法性接口验证，不能只判断范围 `0..186`。

## 7. 各行动的目标解析与验证

### 7.1 防御

不需要目标。直接构建防御命令。

### 7.2 等待

不需要目标。仅当当前单位本回合尚未等待过时允许。

### 7.3 移动

#### 位置目标

- 目标格必须为空；
- 必须在当前单位的可达区域内；
- 双格生物还要验证第二格占用。

#### 部队目标

部队目标只作为“靠近锚点”。流程：

1. 枚举目标部队周围相邻格；
2. 过滤不可达、被占用和双格冲突的格子；
3. 选取移动成本最低的合法格；
4. 构建移动命令。

### 7.4 近战攻击（站立格 + 攻击格）

近战配置采用**模拟玩家点击**模型，而不是“选一个敌方部队再猜邻接方向”：

1. **站立格** `meleeStandHex`：我方单位准备发起攻击时所在/将要站到的格子。未设置时默认使用当前站位。
2. **攻击格** `meleeAttackHex`：模拟鼠标点击的格子。该格上只要有敌人**头格或双格尾格**即视为有效目标。
3. 运行时流程：
   - 若 `meleeAttackHex` 无效 → 失败；
   - 若攻击格上没有敌人（头/尾都不命中）→ 失败（可降级防御）；
   - 否则写：
     - `action = 6`（`WALK_ATTACK`）
     - `actionTarget = 攻击格`
     - `actionParameter = 站立格`
     - 同步 `mouse_coord / attacker_coord / move_type=7`，贴近玩家点选后的状态。
4. UI：近战格子显示两行“站立: Axx / 攻击: Byy”，点击后进入战场选格模式，在面板外点击战场写入 hex。

双格敌人判定使用 `GetSecondSquare (0x4463C0)`；不得自行用六方向猜攻击参数。

### 7.5 远程攻击

- 目标必须是敌方存活部队；
- 若目标来自位置，则位置上必须有敌方部队；
- `self->CanShoot(target)` 必须为真；
- 原版还需处理无弹药、被贴身、失忆、城墙惩罚等状态。

合法后构建远程攻击命令，由原版执行动画与伤害结算。

### 7.6 急救帐篷疗伤

仅适用于：

- 当前活动单位是急救帐篷；
- 当前英雄掌握急救术；
- 目标是己方存活部队；
- 目标确实有可恢复的生命损失；
- 原版急救目标合法性判定通过。

推荐目标策略：

- 指定己方部队；
- 己方损失生命最多；
- 己方随机伤员；
- 己方伤员顺序轮换。

若没有合法伤员：

- 保持原版处理；
- 不对战争机器应用“降级为防御”。

### 7.7 投石车攻击

- 仅适用于投石车且英雄掌握弹道术；
- 目标必须是当前攻城战中的合法城墙段；
- 不允许把普通部队作为投石车目标；
- 城墙段有效性、剩余耐久、弹道术等级对应的目标限制，均复用原版 `FUN_00473530` 的判定路径。

## 8. “允许降级为防御”的最终语义

该勾选项只作用于**普通部队的 H3Auto 主动策略**，不改变战争机器的原版控制权规则。

触发条件：

- 目标策略没有选出合法目标；
- 固定目标死亡、消失或跨战斗指纹不匹配；
- 移动目标不可达；
- 近战找不到合法攻击位置；
- 远程攻击不可射击；
- 动作和目标类型不兼容；
- 主动命令构建失败。

处理规则：

```cpp
if (TryBuildAndSubmitConfiguredAction(...))
    return EXECUTED;

if (rule.allowDefendFallback && IsNormalCombatUnit(self))
    return SubmitDefend(...);

return LEAVE_FOR_HUMAN; // 不交回 AI
```

明确不适用：

- 弹药车；
- 急救帐篷；
- 投石车；
- 弩车；
- 箭塔；
- 手动策略；
- 原版本来就决定交给 AI 的回退分支。

## 9. 控制权决策树

运行时必须有三个结果，而不是简单的布尔值：

```cpp
enum ControlDecision {
    KEEP_ORIGINAL, // 保持原版（可能是人工，也可能是原版特殊逻辑）
    HAND_TO_AI,    // 明确交给原版 AI
    EXECUTE_H3AUTO // H3Auto 构建并提交主动动作
};
```

### 9.1 弹药车

```text
永远 KEEP_ORIGINAL
```

### 9.2 急救帐篷

```text
无急救术：不进入设置列表，KEEP_ORIGINAL
有急救术：
  手动：KEEP_ORIGINAL
  急救治疗：EXECUTE_H3AUTO
```

### 9.3 投石车

```text
无弹道术：不进入设置列表，HAND_TO_AI（等价于原版处理）
有弹道术：
  手动：KEEP_ORIGINAL
  投石车攻击：EXECUTE_H3AUTO
```

### 9.4 弩车与箭塔

```text
配置远程攻击：EXECUTE_H3AUTO（无论是否有炮术）
未配置/手动：
  有炮术：KEEP_ORIGINAL
  无炮术：HAND_TO_AI
```

### 9.5 普通部队

```text
手动：KEEP_ORIGINAL
非手动：EXECUTE_H3AUTO
执行失败：
  允许降级：防御
  不允许降级：留给人类
绝不 HAND_TO_AI
```

## 10. Hook 架构

现有 `FUN_004744D0 @ 0x4744D0` 判定点只能自然表达：

- 保持原版；
- 交给 AI。

完整主动执行需要两个 Hook 分工。

### Hook A：控制权判定

位置：`FUN_004744D0 @ 0x4744D0`。

职责：

1. 调用原函数获取原版判定；
2. 根据当前活动单位、英雄技能和生效方案得到三态 `ControlDecision`；
3. `HAND_TO_AI` 时返回 1；
4. `KEEP_ORIGINAL` 时返回原值；
5. `EXECUTE_H3AUTO` 时记录待提交命令，但不让原版 AI 覆盖目标。

### Hook B：主动动作注入

位置原则：

- 当前活动单位已经确定；
- 原版战争机器特殊预处理已完成；
- 尚未进入 `FUN_004746B0` 的人类输入等待；
- `battle->action == 0`；
- 主循环后续仍会自然进入 `FUN_004786B0`。

具体指令地址需继续从 `FUN_00473A00` 中确认，不能直接用每帧绘制 Hook 替代。

职责：

1. 读取当前 `AutoStackRule`；
2. `ResolveTarget`；
3. 调用原版合法性检查；
4. 构建 `action/actionParameter/actionTarget/actionParameter2`；
5. 若失败且符合降级规则，构建防御动作；
6. 只写一次，随后让原版主循环自然执行。

### 防重入

待提交状态必须绑定：

```text
battle manager + activeStack 指针 + currentMonSide + currentMonIndex
```

当活动单位变化、动作字段不再为 0、战斗结束或面板重新提交方案时清除。禁止在 Hook 内直接调用 `FUN_004786B0`，避免战斗消息循环重入。

## 11. 设置面板 UI 方案（重设计）

旧的“行动下拉 + 目标下拉”两个固定下拉框来自旧的二维模型，无法表达新模型的 5 个维度
（行动 / 目标类型 / 阵营 / 选择器 / 固定目标），且维度间存在依赖，硬塞会产生大量无效组合。
界面改为**渐进式显示（progressive disclosure）**：每格按“行动 → 目标”的因果顺序展开，
后续控件根据前一选择动态出现、隐藏或改变可选项。

### 11.1 设计原则

1. 行动决定后续一切；先选行动，再按行动决定是否需要目标、需要哪种目标。
2. 只呈现合法组合；由第 5 节兼容矩阵驱动控件的显示与选项过滤。
3. 维度依赖内聚在单元格内解决，面板整体框架（5 套方案、滚动、勾选、网格）不变。
4. 无效即隐藏，而非置灰后仍可点。

### 11.2 单元格动态形态

每个己方格子是一张卡片，顶部固定显示单位图标 / 名称 / 位置 / 数量，
下方是**动态控件区**，随行动类型重新布局：

形态 A —— 无目标（手动 / 防御 / 等待）：

```text
┌─ 弩车 A05  ×3 ─────────────┐
│ 行动: [防御 ▾]             │
└────────────────────────────┘
```

形态 B —— 选部队（近战 / 远程 / 急救）：

```text
┌─ 弩车 A05  ×3 ─────────────┐
│ 行动: [远程攻击 ▾]         │
│ 目标: [敌方部队 ▾] [数量最多 ▾] │
│ □ 允许降级为防御           │
└────────────────────────────┘
```

- 阵营下拉按行动过滤：急救只给“己方”，攻击只给“敌方”，移动可给“己方/敌方（作为靠近锚点）”。
- 选择器下拉列出随机/顺序/最近/最远/远程高速/数量高/数量低/伤最重/指定。

### 11.5 状态与生命周期

- 设置面板打开时重建当前战斗的目标目录（部队快照 + 合法位置）。

- 行动改变时立即重算目标区可选项，并把不再合法的目标重置为该行动的默认目标。
- 单元格高度按当前形态自适应：形态 A 最矮，形态 B/C 展开目标行。

面板外层（背景、5 套方案按钮、滚动条、网格金框、OK/取消、勾选提交）保持现有架构不变。
